# Асимптотические обозначения

1. **Big O** ($O$-большое) - верхняя граница. 
	Означает, что некоторая фукнция растет *не быстрее*, чем заданная функция с точностью до константы. Пример: $2 n^2 + 3 n + 1 = O(n^2)$. 
	**Определение**: $f(n) = O(g(n))$, если существуют положительные константы $c$ и $n_0$, такие что $0 \leq f(n) \leq c \cdot g(n)$ для всех $n \geq n_0$.
	
2. **Big Omega** ($\Omega$-большое) - нижняя граница. 
	Означает, что некоторая функция растёт *не медленнее*, чем заданная. Пример: $2 n^ 2 + 3 n + 1 = \Omega(n^2)$.
	**Определение**: $f(n) = \Omega(g(n))$, если существуют положительные константы $c$ и $n_0$, такие что $0 \leq c \cdot g(n) \leq f(n)$ для всех $n \geq n_0$.
	
3. **Big Theta** ($\varTheta$-большое) - точная граница. 
	Означает, что некоторая функция растёт *так же, как* заданная с точностью до константы. 
	**Определение**: $f(n) = \varTheta(g(n))$, если существуют положительные константы $c_1$, $c_2$ и $n_0$, такие что $0 \leq c_1 \cdot g(n) \leq f(n) \leq c_2 \cdot g(n)$ для всех $n \geq n_0$.

4. **Little O** ($o$-малое) - строгая верхняя граница.
	Означает, что функция растёт *строго* медленнее, чем заданная. Пример: $n = o(n^2)$ и $2n^2 \neq o(n^2)$ 
	**Определение**: $f(n) = o(g(n))$, если для любой положительной константы $c > 0$ существует $n_0$, такое что $0 \leq f(n) < c \cdot g(n)$ для всех $n \geq n_0$.

5. **Little Omega** ($\omega$-малое) - строгая нижняя граница.
	Означает, что функция растёт *строго* быстрее, чем заданная. Пример: $n^2 = \omega(n)$
	**Определение**: $f(n) = \omega(g(n))$, если для любой положительной константы $c > 0$ существует $n_0$, такое что $0 \leq c \cdot g(n) < f(n)$ для всех $n \geq n_0$.

**Связь между обозначениями**:
- $f(n) = \varTheta(g(n)) \Leftrightarrow f(n) = O(g(n))$ и $f(n) = \Omega(g(n))$
- $f(n) = o(g(n))$ подразумевает $f(n) = O(g(n))$, но не наоборот
- $f(n) = \omega(g(n))$ подразумевает $f(n) = \Omega(g(n))$, но не наоборот

# Сортировки с использованием сравнений; сортировка вставкой и слиянием; худшие, средние, лучшие оценки выполнения

## Устойчивость в сортировках

**Устойчивость** - свойство алгоритма сортировки сохранять относительный порядок равных элементов. Т.е. два равных элемента, между которыми находится ещё несколько мусорных элементов, в отсортированном списке останутся относительно друг друга на тех же местах, просто мусор уйдёт.

## Сортировка вставкой

- Лучшее время: $O(n)$
- Худшее время: $O(n^2)$
- Среднее время: $O(n^2)$
- Память: $O(1)$ (in-place)
- Устойчивость: Да
```cpp
void insertionSort(int arr[], int n) {
    for (int i = 1; i < n; i++) {
        int key = arr[i]; 
        int j = i - 1;
        while (j >= 0 && arr[j] > key) {
            arr[j + 1] = arr[j];
            j = j - 1;
        }
        arr[j + 1] = key;
    }
}
```

## Сортировка слиянием

**Сортировка слиянием**:
- Лучшее время: $O(n \log n)$
- Худшее время: $O(n \log n)$
- Среднее время: $O(n \log n)$
- Память: $O(n)$ - дополнительный массив
- Устойчивость: Да

**Идея**:
- Взять массив
- Найти середину
- Рекурсивно вызвать себя же для правой части массива
- Рекурсивно вызвать себя же для левой части массива
- Слить получившиеся результаты

**Реализация на C++**:
```cpp
void mergeSort(std::vector<int> &data, int left, int right) {
	if (left < right) {
		int mid = left + ((right - left) / 2);
		mergeSort(data, left, mid);
		mergeSort(data, mid+1, right);
		merge(data, left, mid+1, right);
	}
}

void merge(std::vector<int> &data, int left, int mid, int right) {
	int leftSize = mid - left;
	int rightSize = right - mid + 1;
	std::vector<int> leftArr(leftSize);
	std::vector<int> rightArr(rightSize);
	for (int i = 0; i < leftSize; ++i) {
		leftArr[i] = data[left + i];
	}
	for (int j = 0; j < rightSize; ++j) {
		rightArr[j] = data[mid + j];
	}
	
	int i = 0, j = 0, k = left;
	while (i < leftSize && j < rightSize) {
		if (leftArr[i] <= rightArr[j]) {
			data[k] = leftArr[i];
			++i;
		} else {
			data[k] = rightArr[j];
			++j;
		}
		++k;
	}
	
	while (i < leftSize) {
		data[k] = leftArr[i];
		++i;++k;
	}
	while (j < rightSize) {
		data[k] = rightArr[j];
		++j;++k;
	}
}
```

## Дерево решений

**Дерево решений** — это абстрактная математическая модель, которая описывает работу любого алгоритма сортировки, основанного на сравнениях элементов (Insertion Sort, Merge Sort, Quick Sort и т.д.).

Суть в том, что мы можем представить все сравнения элементов, которые делает алгоритм, в виде бинарного дерева:
- **Внутренние узлы** дерева — это сами проверки. Например, узел `i : j` означает: «Сравниваем $i$-й и $j$-й элементы массива».
- **Ветви** — это результаты сравнения. Левая ветвь означает, например, что $A[i] \leq A[j]$ (ответ "Да"), правая — что $A[i] > A[j]$ (ответ "Нет").
- **Листья** дерева — это итоговый отсортированный массив. Строго говоря, это одна из возможных перестановок исходных элементов $\langle \pi(1), \pi(2), \dots, \pi(n) \rangle$, которая возвращает массив в правильном порядке.

**Почему это важно?**
Любой путь от корня к листу — это сценарий выполнения алгоритма для конкретного входного массива. А **высота дерева** (самый длинный путь от корня до листа) — это максимальное количество сравнений, которое алгоритм сделает в *худшем случае*.

## Теорема о нижней сложностной оценке сортировок сравнением

**Теорема:** Любой алгоритм сортировки, основанный на сравнениях, в худшем случае требует $\Omega(n \log n)$ сравнений. 
*(Другими словами: математически невозможно написать сортировку сравнениями, которая в худшем случае работает быстрее, чем за $n \log n$).*

**Доказательство (по Кормену):**
1. **Сколько существует вариантов на входе?**
   Массив из $n$ элементов можно перемешать $n!$ способами. Алгоритм должен уметь правильно распутать *любую* из этих перестановок.
2. **Сколько должно быть листьев в дереве решений?**
   Чтобы алгоритм работал корректно, каждая возможная перестановка должна приводить к своему правильному ответу. То есть каждая из $n!$ перестановок должна быть представлена хотя бы одним листом в дереве решений. Значит, количество листьев $l$ должно быть не меньше количества перестановок:
   $$l \geq n!$$
3. **Связь листьев и высоты дерева:**
   В любом бинарном дереве высоты $h$ максимальное количество листьев равно $2^h$. Сопоставляя это с предыдущим пунктом, получаем неравенство:
   $$2^h \geq l \geq n!$$
4. **Находим высоту дерева ($h$):**
   Берем логарифм по основанию 2 от обеих частей неравенства:
   $$h \geq \log_2(n!)$$
5. **Оцениваем $\log_2(n!)$:**
   Осталось доказать, что $\log_2(n!) = \Omega(n \log n)$. Для асимптотической оценки факториала используется приближение Стирлинга:
   $$n! \approx \sqrt{2 \pi n} \left( \frac{n}{e} \right)^n$$
   *(Более строго: $n! = \sqrt{2 \pi n} \left( \frac{n}{e} \right)^n \left( 1 + \varTheta\left(\frac{1}{n}\right) \right)$)*
   
   Возьмем логарифм по основанию 2 от обеих частей и воспользуемся свойствами логарифмов (логарифм произведения равен сумме логарифмов, показатель степени выносится как множитель):
   $$\log_2(n!) \approx \log_2\left( \sqrt{2 \pi n} \cdot \left( \frac{n}{e} \right)^n \right) = \log_2\left(\sqrt{2 \pi n}\right) + \log_2\left( \left( \frac{n}{e} \right)^n \right)$$
   Теперь распишем второе слагаемое, так как именно оно растет быстрее всего:
   $$\log_2\left( \left( \frac{n}{e} \right)^n \right) = n \cdot \log_2\left( \frac{n}{e} \right) = n \log_2 n - n \log_2 e$$
   Собираем всё вместе. Первое слагаемое $\log_2\left(\sqrt{2 \pi n}\right)$ растет логарифмически — как $O(\log n)$, что бесконечно мало по сравнению с $n \log n$. В итоге получаем:
   $$\log_2(n!) = n \log_2 n - n \log_2 e + O(\log n)$$
   Отбрасывая константы (такие как $\log_2 e$) и младшие члены ($O(\log n)$), мы видим, что доминирующая часть функции растёт как $n \log n$. Следовательно:
   $$\log_2(n!) = \varTheta(n \log n)$$
   А так как $\varTheta$ подразумевает и верхнюю, и нижнюю границу, то справедливо и $\Omega(n \log n)$.

**Вывод:**
Поскольку высота дерева решений $h \geq \log_2(n!)$, а мы строго доказали, что $\log_2(n!) = \Omega(n \log n)$, то количество сравнений в худшем случае всегда $\geq \Omega(n \log n)$. 

**Следствие теоремы:**
Сортировка слиянием (Merge Sort) или Пирамидальная сортировка (Heap Sort), считаются **асимптотически оптимальными** — их время работы $O(n \log n)$ совпадает с теоретическим нижним пределом $\Omega(n \log n)$. Улучшить эту асимптотику для алгоритмов, основанных на сравнениях, математически невозможно.

# Сортировки за линию

Сортировки подсчётом, поразрядная и карманная не используют сравнения элементов между собой, поэтому на них не распространяется теоретический предел сложности $\Omega(n \log n)$.

## Сортировка подсчётом

### Применение
1. **Тип данных:** Целые числа (или объекты, приводимые к целым ключам).
2. **Диапазон:** Эффективна, когда диапазон значений $k$ сопоставим с количеством элементов $n$ (т.е. $k = O(n)$).
3. **Ограничения:** Требует дополнительную память $O(n + k)$.

### Префиксные суммы

Префиксная сумма в массиве частот — это сумма всех предыдущих счетчиков до текущего индекса включительно.
- **Математически:** $C'[i] = \sum_{j=0}^{i} C[j]$.
- **Смысл:** Значение $C'[i]$ указывает на количество элементов, которые меньше или равны $i$. Это позволяет сразу определить финальную позицию (индекс) элемента в отсортированном массиве.

### Алгоритм (устойчивый)
1. **Инициализация:** Найти $min$ и $max$. Вычислить диапазон $k = max - min + 1$. Создать массив `count` размера $k$ (заполнить нулями) и `res` размера $n$.
2. **Подсчёт:** Пройти по исходному массиву, инкрементируя `count[arr[i] - min]`.
3. **Модификация (Префиксные суммы):** Преобразовать `count` так, чтобы каждый элемент был равен сумме себя и предыдущего: `count[i] = count[i] + count[i-1]`.
4. **Сборка (справа налево):** Пройти по исходному массиву **с конца в начало**. Элемент `arr[i]` поместить в `res` по индексу `count[arr[i] - min] - 1`. Уменьшить значение в `count`.
   * *Проход именно с конца обеспечивает **устойчивость** (последний встреченный одинаковый элемент попадет в самый конец своего блока).*

### Реализация на C++ (устойчивая)

```cpp
std::vector<int> countingSort(const std::vector<int>& data) {
    if (data.empty()) return {};

    // 1. Нахождение диапазона
    auto [minIt, maxIt] = std::minmax_element(data.begin(), data.end());
    int minVal = *minIt;
    int maxVal = *maxIt;
    int k = maxVal - minVal + 1;

    std::vector<int> count(k, 0);
    for (int x : data) {
        count[x - minVal]++;
    }

    for (int i = 1; i < k; ++i) {
        count[i] += count[i - 1];
    }

    std::vector<int> res(data.size());
    for (int i = data.size() - 1; i >= 0; --i) {
        int val = data[i];
        int pos = count[val - minVal] - 1; 
        res[pos] = val;
        count[val - minVal]--; 
    }
    return res;
}
```

### Доказательство линейности и сложности

- **Время:**
    1. Поиск min/max: $O(n)$
    2. Заполнение счетчиков: $O(n)$
    3. Вычисление префиксных сумм: $O(k)$
    4. Расстановка элементов в `res`: $O(n)$
    - **Итого:** $O(n + n + k + n) = O(n + k)$.
- **Память:** $O(k)$ для массива счетчиков + $O(n)$ для результирующего массива = $O(n + k)$.
